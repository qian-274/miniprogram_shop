## 2. 项目背景

社区生鲜团购以小区、社区、自提点为服务半径，用户可在线浏览生鲜商品、参与团购、加入购物车并提交订单，再到指定自提点取货。相比传统线下买菜，社区团购更适合高频、刚需、价格敏感的生鲜消费场景。

本课题拟建设一个“社区生鲜团购管理小程序”，整体设想包含普通用户端、自提点店主端和后台管理端。当前阶段优先完成普通用户端，使用户能够完成从浏览商品到购物车结算、订单查看与个人中心管理的核心闭环。

## 3. 问题陈述

### 3.1 当前情况

社区用户购买生鲜商品时，经常依赖线下菜场、超市或微信群接龙。线下购买存在时间成本高、商品信息不透明、价格对比不方便等问题；微信群团购又存在商品展示混乱、订单统计困难、用户查询订单不便等问题。

### 3.2 用户痛点

- 商品查找效率低：用户难以快速找到水果、蔬菜、肉蛋、粮油等不同品类。
- 团购信息分散：拼团活动、精选商品、优惠商品缺少统一入口。
- 购物车操作不直观：用户需要方便地选择商品、调整数量、查看总价并结算。
- 订单状态不清晰：用户需要随时查看待发货、待收货、待评价、已完成等订单状态。
- 服务入口不统一：收藏、客服、反馈、退出登录等个人功能需要集中管理。

### 3.3 业务影响

- 提升社区用户购买生鲜商品的效率和体验。
- 提高团购商品曝光率，促进拼团抢购和今日精选商品转化。
- 规范购物车与订单流程，降低订单遗漏和沟通成本。
- 为后续自提点店主端、后台管理端提供清晰的业务边界。

## 4. 产品目标

### 4.1 核心目标

普通用户端需要支持用户完成以下核心流程：

1. 进入首页浏览推荐商品、商品分类、拼团抢购与今日精选。
2. 通过搜索或分类快速找到目标商品。
3. 查看商品信息并加入购物车。
4. 在购物车中选择商品、调整数量、查看总价并发起结算。
5. 在“我的”页面查看订单状态、收藏、客服、反馈和账号相关功能。

### 4.2 成功指标

- 用户可在 3 次点击以内进入任意一级商品分类。
- 用户可在购物车页面直接完成勾选、数量调整、总价确认和结算操作。
- 首页能清晰呈现搜索、轮播图、分类入口、拼团抢购和今日精选模块。
- “我的”页面能清晰呈现订单状态入口和常用个人服务入口。
- 主要页面视觉风格统一，符合生鲜团购绿色、清爽、便捷的产品感受。

## 5. 用户角色

### 5.1 普通用户

普通用户是本阶段的主要使用者，通常为社区居民、家庭采购者或日常生鲜消费者。

核心诉求：

- 快速发现新鲜、优惠、适合家庭采购的商品。
- 清楚了解商品价格、图片、名称、规格等信息。
- 能方便地加入购物车并合并结算。
- 能查看订单进度和个人服务入口。

### 5.2 自提点店主

自提点店主是后续阶段角色，本阶段仅作为业务边界存在。普通用户端需要让用户理解订单最终将由自提点或平台完成履约，但不定义店主端具体功能。

### 5.3 后台管理员

后台管理员是后续阶段角色，本阶段仅作为商品、订单、轮播图、分类等内容的管理来源，不定义后台管理端具体功能。

## 8. 核心业务流程

### 8.1 浏览购买流程

1. 用户进入小程序首页。
2. 用户通过搜索框、分类入口、拼团抢购或今日精选进入商品列表或商品详情。
3. 用户查看商品信息。
4. 用户将商品加入购物车。
5. 用户进入购物车，选择需要购买的商品。
6. 用户调整商品数量并确认合计金额。
7. 用户点击结算，进入订单确认流程。
8. 用户提交订单后，可在“我的”页面查看订单状态。

### 8.2 购物车管理流程

1. 用户进入购物车。
2. 页面展示已加入购物车的商品。
3. 用户可单选、多选或全选商品。
4. 用户可增加或减少商品数量。
5. 页面根据已选商品实时展示合计金额。
6. 用户可删除不需要的商品。
7. 用户点击结算进入下一步。

### 8.3 个人中心流程

1. 用户进入“我的”页面。
2. 页面展示用户头像、昵称和订单状态入口。
3. 用户点击订单状态查看对应订单。
4. 用户可进入我的收藏、在线客服、意见反馈。
5. 用户可执行退出登录。

## 9. 功能需求

### 9.1 底部 TabBar

底部导航包含“首页”“分类”“购物车”“我的”四个入口。

功能要求：

- 当前选中的 Tab 需要有明显高亮状态。
- 未选中的 Tab 以弱化样式展示。
- 用户点击任一 Tab 后，应切换到对应页面。
- Tab 顺序固定为：首页、分类、购物车、我的。
- 视觉风格参考截图，主色调以绿色为核心，强调生鲜、健康、便捷。

验收标准：

- 用户能从任意一级页面切换到其他 Tab 页面。
- 当前页面对应 Tab 的图标和文字均有选中状态。
- 四个 Tab 名称与顺序保持一致。

### 9.2 首页

首页是普通用户端的主要流量入口，布局参考图 1，自上而下包含商品搜索框、轮播图、商品简单分类、拼团抢购、今日精选。

#### 9.2.1 顶部区域

功能要求：

- 页面顶部展示小程序标题，建议名称为“农产品拼团商城”或与课题一致的产品名称。
- 顶部主色为绿色，整体保持清爽、醒目的视觉风格。
- 顶部区域下方展示商品搜索框。

验收标准：

- 用户进入首页后可立即看到标题和搜索框。
- 搜索框占据页面上方主要视觉位置。

#### 9.2.2 商品搜索框

功能要求：

- 搜索框提示文案建议为“请输入商品关键词”。
- 用户点击搜索框后，可输入商品名称、分类关键词或商品相关词。
- 提交搜索后展示与关键词相关的商品结果。
- 当搜索结果为空时，需要给出空状态提示。

验收标准：

- 输入有效关键词后能进入搜索结果页或展示搜索结果。
- 输入无匹配关键词时能看到明确提示。
- 搜索框在首页顶部固定位置清晰可见。

#### 9.2.3 轮播图

功能要求：

- 轮播图用于展示生鲜团购活动、主推商品、节日促销或平台公告。
- 轮播图应位于搜索框下方。
- 轮播图支持多张图片轮换展示。
- 用户点击轮播图，可进入对应活动或商品详情。
- 轮播图下方展示当前轮播位置指示。

验收标准：

- 首页轮播图能正常展示至少一张图片。
- 多张轮播内容之间可切换。
- 点击轮播图能进入对应内容页面。

#### 9.2.4 商品简单分类

首页分类区参考图 1，以图标加文字形式展示常用分类。

建议分类：

- 新鲜水果
- 时令蔬菜
- 禽畜肉类
- 蛋品采购
- 粮油调味
- 新鲜乳品
- 五谷杂粮
- 干货菌菇

功能要求：

- 分类以宫格方式展示，建议每行四个。
- 每个分类包含分类图标和分类名称。
- 用户点击分类后进入对应分类商品列表。
- 分类名称需要简短、易懂，避免换行过多。

验收标准：

- 首页能展示不少于 8 个常用分类入口。
- 点击任一分类后能看到对应分类商品。
- 分类样式在页面中整齐排列。

#### 9.2.5 拼团抢购

拼团抢购用于展示限时或重点团购商品。

功能要求：

- 拼团抢购模块位于首页分类区下方。
- 模块标题为“拼团抢购”。
- 商品卡片展示商品图片、名称、价格、团购标识或活动信息。
- 用户点击商品卡片进入商品详情。
- 商品售罄或活动结束时，需要展示对应状态。

验收标准：

- 首页能看到“拼团抢购”模块标题。
- 模块内至少可展示一个商品卡片。
- 商品卡片信息能帮助用户判断是否购买。

#### 9.2.6 今日精选

今日精选用于展示平台推荐商品、应季商品或高性价比商品。

功能要求：

- 模块标题为“今日精选”。
- 商品卡片展示图片、名称、价格、销量或推荐理由。
- 用户点击商品卡片进入商品详情。
- 今日精选可与拼团抢购形成区分，避免内容重复感过强。

验收标准：

- 首页能看到“今日精选”模块。
- 商品信息展示完整清晰。
- 点击商品后能进入商品详情或商品购买路径。

### 9.3 分类页

分类页用于系统化展示所有商品品类，帮助用户按需求浏览商品。

功能要求：

- 分类页展示一级商品分类。
- 用户选择某一分类后，展示该分类下的商品列表。
- 商品列表应展示商品图片、名称、价格、简要描述和购买入口。
- 分类页应支持从分类进入商品详情。
- 当某分类暂无商品时，应展示空状态提示。

验收标准：

- 用户能通过分类页浏览不同商品品类。
- 切换分类后商品列表内容随之变化。
- 商品卡片信息清晰，不影响用户判断。

### 9.4 商品详情页

商品详情页用于承载用户购买决策。

功能要求：

- 展示商品图片、商品名称、价格、规格、库存状态、商品描述。
- 展示商品是否参与拼团、是否为精选商品等活动信息。
- 用户可选择购买数量。
- 用户可将商品加入购物车。
- 用户可直接进入结算流程。
- 用户可收藏商品。
- 当商品售罄或不可购买时，购买入口需要置灰或提示不可购买原因。

验收标准：

- 用户能从首页、分类页、搜索结果或购物车进入商品详情。
- 商品详情信息完整，能支撑用户购买决策。
- 加入购物车后购物车数量和内容应更新。

### 9.5 购物车页

购物车页布局与功能参考图 2，主要包含商品列表、选择状态、数量调整、合计金额和结算入口。

#### 9.5.1 商品列表

功能要求：

- 展示用户已加入购物车的商品。
- 每个商品行展示选择按钮、商品图片、商品名称、商品价格、购买数量、增加按钮、减少按钮。
- 商品名称过长时需要合理展示，不应影响价格和数量操作。
- 商品失效、售罄或库存不足时，需要给出提示并限制结算。

验收标准：

- 购物车有商品时能按列表展示商品。
- 商品图片、名称、价格、数量、选择状态清晰可见。
- 商品不可购买时有明确状态提示。

#### 9.5.2 商品选择

功能要求：

- 用户可勾选或取消勾选单个商品。
- 页面底部提供“全选”入口。
- 全选后所有可购买商品均被选中。
- 取消全选后所有商品取消选中。

验收标准：

- 单个商品选择状态可正常切换。
- 全选状态与商品选择状态保持一致。
- 不可购买商品不应参与结算合计。

#### 9.5.3 数量调整

功能要求：

- 用户可点击加号增加商品数量。
- 用户可点击减号减少商品数量。
- 数量不能小于 1。
- 数量达到库存上限或购买限制时，应提示用户。
- 数量变化后，合计金额应同步更新。

验收标准：

- 点击加号后商品数量增加。
- 点击减号后商品数量减少。
- 数量变化后合计金额正确变化。
- 数量达到限制时有清晰提示。

#### 9.5.4 合计与结算

功能要求：

- 底部固定展示全选、合计金额和结算按钮。
- 合计金额只计算已选中的可购买商品。
- 合计金额格式清晰，突出价格信息。
- 未选择商品时，结算按钮应不可用或点击后提示选择商品。
- 点击结算后进入订单确认流程。

验收标准：

- 勾选商品后合计金额正确展示。
- 未勾选商品时不能直接结算。
- 点击结算能进入订单确认流程。

#### 9.5.5 空购物车

功能要求：

- 当购物车为空时，展示空状态。
- 空状态应提供返回首页或去逛逛入口。

验收标准：

- 无商品时不展示空白页面。
- 用户能从空购物车快速回到商品浏览路径。

### 9.6 订单确认页

订单确认页用于用户提交订单前的最终确认。

功能要求：

- 展示待购买商品列表、数量、单价和小计。
- 展示订单总金额。
- 展示自提点或取货信息入口。
- 展示用户联系方式入口。
- 支持用户填写订单备注。
- 用户确认无误后可提交订单。

验收标准：

- 用户能确认商品、数量、金额和取货信息。
- 必填信息缺失时不能提交订单，并给出提示。
- 订单提交成功后给出成功反馈，并引导用户查看订单。

### 9.7 订单状态

订单状态入口主要在“我的”页面展示，参考图 3。

功能要求：

- 展示四个订单状态入口：待发货、待收货、待评价、已完成。
- 用户点击某个状态后，进入对应订单列表。
- 订单列表展示订单商品、订单金额、订单状态、下单时间、操作入口。
- 不同状态下展示不同可用操作，例如确认收货、去评价、查看详情。

验收标准：

- “我的”页面能清晰看到四个订单状态。
- 点击状态后能进入对应订单列表。
- 空订单状态需要展示友好的空状态提示。

### 9.8 我的页面

我的页面布局与功能参考图 3，主要承载个人信息、订单入口和常用服务。

#### 9.8.1 用户信息

功能要求：

- 页面顶部展示用户头像和昵称。
- 未登录状态下应引导用户登录或授权。
- 登录后展示用户昵称。

验收标准：

- 用户进入“我的”页面后能看到个人信息区域。
- 未登录用户能看到登录引导。

#### 9.8.2 订单快捷入口

功能要求：

- 展示待发货、待收货、待评价、已完成四个入口。
- 每个入口使用图标加文字形式展示。
- 如存在对应状态订单，可展示数量提醒。

验收标准：

- 四个订单入口名称准确。
- 点击入口后进入对应订单列表。
- 订单状态图标视觉统一。

#### 9.8.3 我的收藏

功能要求：

- 用户可进入我的收藏列表。
- 收藏列表展示用户收藏过的商品。
- 用户可从收藏列表进入商品详情。
- 用户可取消收藏。

验收标准：

- 用户能查看已收藏商品。
- 取消收藏后列表状态更新。

#### 9.8.4 在线客服

功能要求：

- 用户可通过在线客服入口联系平台或商家。
- 客服入口需要放在“我的”页面常用服务区域。

验收标准：

- 点击在线客服后能进入客服沟通或客服引导页面。
- 若客服不可用，需要展示替代联系方式或提示。

#### 9.8.5 意见反馈

功能要求：

- 用户可提交使用问题、商品建议、服务建议或其他反馈。
- 反馈内容应支持文字描述。
- 提交成功后需要给出反馈成功提示。

验收标准：

- 用户能进入反馈页面并填写内容。
- 空内容不能提交。
- 提交后用户能看到明确结果反馈。

#### 9.8.6 退出登录

功能要求：

- 用户可点击退出登录。
- 退出前需要二次确认。
- 退出后回到未登录状态。

验收标准：

- 点击退出登录后出现确认提示。
- 确认退出后个人信息变为未登录状态。

## 10. 页面视觉与交互要求

### 10.1 视觉风格

- 主色调以绿色为主，传达生鲜、健康、自然、便捷的品牌感。
- 背景以白色或浅色为主，突出商品图片与价格。
- 价格信息使用醒目颜色，便于用户快速识别。
- 页面整体应保持清爽，不堆叠过多装饰元素。
- 图标风格统一，分类图标应与商品品类强相关。

### 10.2 布局要求

- 首页布局顺序为：顶部标题、搜索框、轮播图、商品分类、拼团抢购、今日精选。
- 购物车底部结算栏固定展示，便于用户随时结算。
- “我的”页面订单状态入口应位于用户信息下方，服务入口按列表排列。
- 页面内容需要适配常见手机屏幕，不应出现文字遮挡或按钮难以点击。

### 10.3 交互要求

- 所有可点击元素需要有清晰的点击反馈。
- 商品加入购物车、提交订单、收藏、取消收藏等关键操作需要结果提示。
- 删除、退出登录等可能造成状态变化的操作需要确认。
- 空状态、加载状态、错误状态需要有明确文案。

### 11.2 订单基础信息

- 订单金额

### 11.3 用户基础信息

用户展示至少应包含：

- 头像

- 订单状态入口

## 12. 用户故事与验收标准

### 12.1 首页浏览

用户故事：

作为普通用户，我希望进入小程序后可以快速看到搜索、活动、分类和推荐商品，以便快速找到想买的生鲜商品。

验收标准：

- 首页首屏可看到搜索框、轮播图和分类入口。
- 用户可通过分类、拼团抢购、今日精选进入商品购买路径。

### 12.2 商品搜索

用户故事：

作为普通用户，我希望通过关键词搜索商品，以便快速找到目标商品。

验收标准：

- 搜索框支持输入关键词。
- 搜索后展示相关商品。
- 无结果时展示空状态。

### 12.3 分类浏览

用户故事：

作为普通用户，我希望按照水果、蔬菜、肉类、蛋品等分类浏览商品，以便提高挑选效率。

验收标准：

- 分类入口清晰可见。
- 点击分类后展示对应商品列表。
- 商品列表可进入商品详情。

### 12.4 加入购物车

用户故事：

作为普通用户，我希望在浏览商品时能把想买的商品加入购物车，以便统一结算。

验收标准：

- 商品详情页提供加入购物车入口。
- 加入成功后给出提示。
- 购物车页面能看到新增商品。

### 12.5 购物车结算

用户故事：

作为普通用户，我希望在购物车中勾选商品、调整数量并查看合计金额，以便确认购买内容后结算。

验收标准：

- 支持单选、多选、全选。
- 支持数量增加和减少。
- 合计金额随选择和数量变化更新。
- 点击结算进入订单确认流程。

### 12.6 查看订单

用户故事：

作为普通用户，我希望在个人中心按状态查看订单，以便了解购买进度。

验收标准：

- “我的”页面展示待发货、待收货、待评价、已完成。
- 点击任一状态后展示对应订单列表。
- 无订单时展示空状态。

### 12.7 使用个人服务

用户故事：

作为普通用户，我希望在“我的”页面找到收藏、客服、反馈和退出登录入口，以便管理个人信息和服务需求。

验收标准：

- “我的”页面展示我的收藏、在线客服、意见反馈、退出登录。
- 点击每个入口能进入对应功能。
- 退出登录前有确认提示。

## 13. 业务规则

### 13.1 商品规则

- 商品应归属到明确分类。
- 商品不可购买时，应明确展示状态。
- 商品价格需要在列表、详情、购物车和订单中保持一致。
- 拼团商品和普通商品需要在展示上有所区分。

### 13.2 购物车规则

- 购物车只展示用户已加入的商品。
- 购物车合计金额只计算已选商品。
- 不可购买商品不参与结算。
- 商品数量不能小于 1。
- 当商品数量达到限制时，继续增加应提示用户。

### 13.3 订单规则

- 提交订单前必须确认商品、数量、金额和取货信息。
- 订单提交后进入待发货状态。
- 用户确认收货后，订单可进入待评价状态。
- 用户评价完成或流程结束后，订单进入已完成状态。

### 13.4 账号规则

- 未登录用户可浏览商品。
- 涉及加入购物车、收藏、提交订单、查看订单等个人行为时，应引导登录。
- 退出登录后，不再展示当前用户的个人信息。

## 14. 异常与空状态

| 场景 | 展示要求 |

| 购物车为空 | 展示空购物车提示，并提供去首页逛逛入口 |
| 商品售罄 | 商品展示售罄状态，限制购买 |
| 网络或加载异常 | 展示加载失败提示，并提供重试入口 |
| 未登录访问个人功能 | 引导用户登录 |

## 15. 质量要求

- 页面结构清晰，用户能理解每个区域的用途。
- 关键购买流程完整，不出现流程中断。
- 页面跳转路径明确，用户能从任意核心页面回到首页或底部 Tab。

- 常见空状态和异常状态不应出现无提示的空白页面。

## 21. 团购功能补充需求 v1.1

### 21.1 需求背景

当前普通用户端已经支持商品浏览、商品详情、购物车结算和订单查看，但“拼团抢购”主要停留在展示层，用户无法真正完成开团、参团、邀请好友、按团购价支付以及成团后的订单流转。为了让社区生鲜团购场景更完整，需要在商品详情页增加真实可操作的团购流程。

本次团购功能参考提供的商品详情截图，核心表现为：商品正常价格下方展示团购价，用户可点击“立即开团”；开团前先选择自提点，随后按参团价提交订单并支付；支付后商品详情页展示该商品当前未满员拼团；用户自己开的团展示“邀请”入口，别人开的团展示团购价参团入口；每个团满 5 人即可享受团购价，满员后不再展示在该商品的可参团列表中。

### 21.2 产品目标

- 让用户在商品详情页清楚看到普通价和团购价差异。
- 支持用户发起 5 人团，并通过微信好友邀请提升成团率。
- 支持用户直接加入他人未满员拼团，并按团购价提交订单。
- 将团购订单纳入现有订单体系，形成“拼团中、待发货、待收货、待评价、已完成”的购买闭环。
- 保证满员拼团从商品详情页的可参团列表中自动隐藏，避免用户加入已满团。

### 21.3 本期范围

本期新增以下功能：

- 商品详情页展示普通价、团购价、库存、分享入口和“立即开团”按钮。
- 商品详情页展示当前商品未满员拼团列表。
- 用户可点击“立即开团”，先选择自提点，再进入订单确认页。
- 用户开团订单按团购价计算金额并完成支付。
- 支付成功后创建拼团，拼团人数初始包含开团用户本人。
- 用户自己开的团在商品详情页展示“邀请”按钮，可邀请微信好友。
- 用户可加入别人创建且未满 5 人的拼团，按钮展示团购价和“去参团”文案。
- 参团用户按团购价提交订单并支付。
- 拼团满 5 人后自动成团，相关订单进入待发货状态。
- 满员拼团不再显示在商品详情页的现有拼团列表中。
- 我的订单中支持查看拼团订单状态和拼团进度。

本期不包含以下功能：

- 复杂拼团营销玩法，如阶梯团价、团长佣金、团长免单、限时秒杀排队。
- 跨商品组合拼团。
- 优惠券与团购价叠加规则。
- 后台团购活动配置页面。
- 完整售后退款审核流程。本期仅定义超时未成团的自动退款或取消规则，具体支付退款接口可在后续实现。

### 21.4 角色与术语

| 术语 | 定义 |
| --- | --- |
| 开团用户 | 第一个按团购价下单并支付，创建拼团的用户 |
| 参团用户 | 加入某个未满员拼团并支付的用户 |
| 拼团价 | 商品参与拼团时的优惠价格，低于普通价 |
| 普通价 | 商品非拼团购买时的正常价格 |
| 成团人数 | 本期固定为 5 人 |
| 拼团中 | 已支付但未满 5 人的拼团状态 |
| 已成团 | 拼团达到 5 人，用户享受团购价，订单进入履约流程 |
| 已满团 | 拼团人数已满 5 人，不再允许新用户加入 |
| 自提点 | 拼团订单绑定的取货地点，开团时必须选择 |

### 21.5 核心业务规则

#### 21.5.1 价格规则

- 商品详情页必须同时展示普通价和团购价。
- 普通价位于商品名称附近或主价格区，团购价位于普通价下方的醒目区域。
- 团购价展示文案建议为“拼团价：¥xx”。
- 用户开团或参团时，订单商品单价必须使用团购价。
- 若商品不参与团购，则不展示团购价、“立即开团”和现有拼团列表。
- 团购价不得高于普通价。

#### 21.5.2 开团规则

- 用户点击“立即开团”后，需要先选择自提点。
- 自提点选择完成后，进入订单确认页。
- 订单确认页展示商品、团购价、购买数量、自提点、联系方式、订单备注和应付金额。
- 用户支付成功后，系统创建一个新的拼团。
- 新拼团人数初始为 1 人，即开团用户本人。
- 开团用户的订单状态为“拼团中”。
- 开团成功后，返回商品详情页时，该拼团应展示在现有拼团列表中。
- 开团用户看到自己创建的拼团时，按钮文案为“邀请”，用于邀请微信好友参团。

#### 21.5.3 参团规则

- 商品详情页展示当前商品所有未满 5 人且仍有效的拼团。
- 用户看到别人创建的拼团时，应展示剩余成团人数，例如“还差 3 人拼成”。
- 参团按钮展示团购价，建议文案为“¥18.8 去参团”。
- 用户点击参团后，进入订单确认页。
- 参团订单使用该拼团绑定的自提点；若后续需要支持用户选择不同自提点，应视为创建新团，不加入当前团。
- 用户支付成功后加入该拼团，拼团人数加 1。
- 同一用户不能重复加入同一个拼团。
- 若用户点击参团时拼团已满，应提示“该拼团已满员，请选择其他拼团或重新开团”。

#### 21.5.4 成团规则

- 本期每个拼团固定 5 人成团。
- 当拼团支付成功人数达到 5 人时，拼团状态变为“已成团”。
- 已成团后，该拼团下所有订单状态由“拼团中”变为“待发货”。
- 已成团拼团不再显示在商品详情页的“现有拼团”列表中。
- 已成团拼团仍可在订单详情中查看拼团成员和成团时间。
- 若商品库存不足以支撑剩余参团人数，应限制继续参团，并提示库存不足。

#### 21.5.5 超时与失败规则

- 拼团应设置有效期，建议本期默认为 24 小时。
- 超过有效期仍未满 5 人时，拼团状态变为“未成团”。
- 未成团订单应进入“已取消”或“退款中/已退款”状态。
- 本期毕业设计演示版可先模拟支付成功、退款成功和状态流转，不强制接入真实微信支付退款接口。
- 若支付失败，则不创建拼团成员记录，不占用拼团名额。

### 21.6 页面与交互需求

#### 21.6.1 商品详情页

商品详情页新增团购区域，页面结构建议如下：

1. 商品主图。
2. 普通价、库存、商品名称、分享入口。
3. 团购价区域，展示“拼团价：¥xx”和“立即开团”按钮。
4. 现有拼团列表，展示当前未满员拼团。
5. 商品推荐或商品详情内容。
6. 底部操作栏保留加入购物车、立即购买等入口。

功能要求：

- 普通价与团购价视觉层级清晰，团购价区域需要突出优惠感。
- “立即开团”按钮固定展示在团购价区域右侧或下方。
- 点击“立即开团”前，如果用户未登录，需要引导登录。
- 商品售罄时，“立即开团”和“去参团”按钮置灰。
- 商品不支持团购时，不展示团购区域。
- 现有拼团列表只展示未满员且未过期拼团。
- 列表中每个拼团至少展示团长头像、团长昵称、剩余人数、操作按钮。
- 自己开的团显示“邀请”按钮。
- 别人开的团显示“¥xx 去参团”按钮。
- 若当前商品没有可参与拼团，展示“暂无可参与拼团，立即开团邀请好友”类提示。

验收标准：

- 用户进入支持团购的商品详情页后，能看到普通价和团购价。
- 点击“立即开团”能进入自提点选择流程。
- 开团支付成功后，商品详情页能看到自己创建的拼团。
- 自己创建的拼团按钮为“邀请”。
- 别人创建的拼团按钮为团购价参团入口。
- 拼团满 5 人后，该拼团不再出现在商品详情页可参团列表中。

#### 21.6.2 自提点选择页或弹窗

功能要求：

- 开团前必须选择自提点。
- 自提点列表展示自提点名称、地址、可取货时间和距离信息。
- 用户选择自提点后进入订单确认页。
- 若用户已有默认自提点，可默认选中，但仍允许切换。
- 若没有可用自提点，需要提示“暂无可用自提点”并阻止继续开团。

验收标准：

- 用户点击“立即开团”后能看到自提点选择入口。
- 用户选择自提点后能继续提交订单。
- 未选择自提点时不能进入支付。

#### 21.6.3 订单确认页

功能要求：

- 团购订单确认页需要明确标识“拼团订单”。
- 商品单价按团购价展示。
- 应付金额按团购价乘以购买数量计算。
- 页面展示绑定的自提点、联系方式、备注和支付按钮。
- 开团订单提交按钮建议为“支付并开团”。
- 参团订单提交按钮建议为“支付并参团”。

验收标准：

- 开团和参团订单均按团购价计算。
- 订单确认页能看到自提点信息。
- 支付成功后能进入对应拼团状态。

#### 21.6.4 支付结果与拼团状态

功能要求：

- 支付成功后，如果是开团，提示“开团成功，邀请好友参团”。
- 支付成功后，如果是参团，提示“参团成功，还差 x 人成团”或“参团成功，拼团已成”。
- 支付失败时，提示支付失败并允许重新支付。
- 拼团中订单应展示剩余成团人数。
- 已成团订单应展示成团成功标识和待发货状态。

验收标准：

- 支付成功后能看到明确结果反馈。
- 拼团人数变化正确。
- 满 5 人后订单状态能自动变为待发货。

#### 21.6.5 邀请好友

功能要求：

- 自己创建的拼团显示“邀请”按钮。
- 点击“邀请”后调用微信分享能力，支持分享给微信好友。
- 分享内容应包含商品名称、团购价、剩余人数和自提点简要信息。
- 好友通过分享进入后，应打开对应商品详情页并定位到对应拼团。
- 若好友打开时拼团已满，需要提示“该拼团已满员”，并展示其他可参与拼团或“立即开团”入口。

验收标准：

- 自己开的团能看到邀请按钮。
- 邀请路径能带上商品和拼团标识。
- 好友进入后可直接参与未满员拼团。

#### 21.6.6 我的订单

功能要求：

- 我的订单需要支持展示“拼团中”订单。
- 拼团中订单展示商品、团购价、已参团人数、还差人数、自提点和邀请入口。
- 已成团订单进入待发货状态。
- 未成团订单展示取消或退款状态。
- 用户可从订单进入商品详情或拼团详情。

验收标准：

- 开团或参团成功后，用户能在我的订单中看到拼团中订单。
- 满 5 人后订单状态变为待发货。
- 未成团订单有明确状态提示。

### 21.7 数据展示要求

#### 21.7.1 商品团购信息

商品至少需要包含以下团购字段：

- 是否支持团购。
- 普通价。
- 团购价。
- 成团人数，默认 5。
- 团购库存或可参与库存。
- 团购活动状态。
- 团购有效期规则。

#### 21.7.2 拼团信息

拼团至少需要包含以下字段：

- 拼团编号。
- 商品编号。
- 团长用户编号。
- 自提点编号。
- 拼团价。
- 成团人数。
- 当前已支付人数。
- 剩余人数。
- 拼团状态：拼团中、已成团、未成团、已满员、已取消。
- 创建时间。
- 过期时间。
- 成团时间。

#### 21.7.3 拼团成员信息

拼团成员至少需要包含以下字段：

- 用户编号。
- 用户头像。
- 用户昵称。
- 是否团长。
- 支付状态。
- 加入时间。
- 关联订单编号。

#### 21.7.4 团购订单信息

团购订单至少需要包含以下字段：

- 订单编号。
- 商品编号。
- 拼团编号。
- 用户编号。
- 购买数量。
- 商品普通价。
- 商品团购价。
- 实付金额。
- 自提点信息。
- 支付状态。
- 订单状态。
- 下单时间。

### 21.8 关键用户故事

#### 21.8.1 用户立即开团

作为普通用户，我希望在商品详情页看到团购价并点击“立即开团”，选择自提点后按团购价支付，从而创建自己的拼团并邀请好友。

验收标准：

- 商品详情页展示普通价和团购价。
- 点击“立即开团”进入自提点选择。
- 选择自提点后进入订单确认。
- 订单金额按团购价计算。
- 支付成功后创建拼团，详情页显示邀请按钮。

#### 21.8.2 用户加入他人拼团

作为普通用户，我希望在商品详情页看到别人正在开的团，并直接按拼团价参团，以便更快享受团购价。

验收标准：

- 商品详情页显示未满员拼团列表。
- 别人的拼团展示剩余人数和团购价参团按钮。
- 点击参团后按团购价下单支付。
- 支付成功后拼团人数增加。
- 满 5 人后拼团成功。

#### 21.8.3 用户邀请微信好友

作为开团用户，我希望支付开团后可以邀请微信好友参团，以便尽快满 5 人成团。

验收标准：

- 自己开的团展示邀请按钮。
- 点击邀请能触发微信好友分享。
- 好友打开分享后能进入对应商品和拼团。
- 拼团已满时好友不能继续加入。

#### 21.8.4 用户查看拼团订单

作为普通用户，我希望在我的订单中查看拼团状态，以便知道还差几人成团、是否已发货或是否未成团退款。

验收标准：

- 拼团中订单显示还差人数。
- 成团后订单变为待发货。
- 未成团订单展示取消或退款提示。

### 21.9 异常与边界场景

| 场景 | 处理规则 |
| --- | --- |
| 用户未登录点击立即开团 | 引导用户登录后再开团 |
| 用户未选择自提点 | 阻止提交订单并提示请选择自提点 |
| 商品售罄 | 禁用立即开团和参团入口 |
| 拼团已满 | 禁止加入，提示该拼团已满员 |
| 拼团过期 | 不再展示在商品详情页，并将订单置为未成团 |
| 支付失败 | 不创建拼团成员，不占用名额，允许重新支付 |
| 用户重复加入同一团 | 阻止重复参团，提示已参与该拼团 |
| 库存不足 | 禁止开团或参团，提示库存不足 |
| 分享进入时拼团已满 | 展示已满员提示，并推荐其他拼团或开新团 |

### 21.10 视觉与交互要求

- 团购价区域使用浅绿色背景，按钮使用红橙渐变或醒目红色，突出优惠和行动感。
- 普通价使用较大字号展示，团购价可使用更强对比色。
- “立即开团”“去参团”“邀请”三个按钮需要视觉区分：
  - 立即开团：强调主行动。
  - 去参团：展示团购价，强调价格吸引。
  - 邀请：强调社交传播。
- 拼团列表需要控制高度，若拼团较多，可展示前 2-3 个并提供“查看更多”。
- 拼团成员头像、昵称、剩余人数文案不能遮挡价格和按钮。
- 页面整体继续保持绿色生鲜商城风格，与现有首页、分类页、购物车和我的页面一致。

### 21.11 技术与实现约束

- 正式版支付应接入微信支付；毕业设计演示版可先使用模拟支付成功流程。
- 微信好友邀请应使用小程序分享能力，分享路径需携带商品编号和拼团编号。
- 拼团人数统计必须以支付成功人数为准，未支付订单不计入成团人数。
- 拼团状态变化需要同步更新商品详情页、订单列表和订单详情。
- 为避免并发超员，参团支付成功前需要校验拼团是否仍有名额。
- 若接入真实后端，需要在服务端完成拼团名额锁定、支付回调、成团状态变更和退款处理。

### 21.12 成功指标

- 用户能在商品详情页 1 次点击进入开团流程。
- 用户能在 3 步内完成开团：立即开团、选择自提点、支付并开团。
- 用户能在商品详情页直接识别自己开的团和别人开的团。
- 成团人数达到 5 人后，拼团状态正确变为已成团。
- 已满团拼团不再出现在商品详情页可参团列表。
- 团购订单金额与团购价保持一致。

### 21.13 验收清单

- 商品详情页展示普通价、团购价和立即开团按钮。
- 点击立即开团后先选择自提点。
- 自提点选择后进入订单确认页。
- 开团订单按团购价计算并支付。
- 支付成功后创建拼团，人数为 1/5。
- 自己创建的拼团展示邀请按钮。
- 邀请按钮可触发微信好友分享。
- 别人创建的拼团展示“团购价 + 去参团”按钮。
- 参团支付成功后拼团人数增加。
- 拼团满 5 人后变为已成团。
- 已成团拼团不再显示在商品详情页现有拼团列表。
- 我的订单中可查看拼团中订单。
- 成团后订单进入待发货状态。
- 未登录、未选择自提点、售罄、拼团已满、支付失败等异常场景均有明确提示。

### 21.14 待确认问题

以下规则建议在开发前确认：

- 拼团有效期是否固定为 24 小时，还是按商品配置。
- 未成团后的处理方式是自动退款、取消订单，还是允许用户补差价转普通购买。
- 参团用户是否必须使用开团用户选择的同一自提点。本 PRD 暂定同一拼团绑定同一自提点。
- 单个用户是否允许为同一商品同时创建多个未满员拼团。本 PRD 暂定不允许。
- 团购商品是否允许加入购物车。本 PRD 暂定团购价只能通过开团或参团下单，购物车仍按普通商品逻辑处理。

## 22. 店主端与可视化后台技术方案 v1.2

### 22.1 背景与目标

当前普通用户端已经具备商品浏览、购物车、下单、订单查看和拼团流程。下一阶段需要补齐自提点店主端和后台管理端，使系统从“用户演示版”升级为“可运营的生鲜团购系统”。

本阶段目标：

- 自提点店主能够查看自己门店相关的到店商品、订单和收货人信息。
- 自提点店主能够维护自提点基础信息，如地址、营业时间、联系电话和公告。
- 后台管理员能够通过可视化后台管理首页轮播图、商品、商品推荐、价格、团购价、自提点和订单。
- 将现有前端 mock 数据逐步迁移到云数据库，为多用户、多店主、多订单场景提供真实数据支撑。
- 在不引入自建服务器的前提下，降低开发、部署和维护成本，适合小程序开发新手完成毕业设计与演示。

### 22.2 推荐技术路线

本项目推荐采用“单小程序多角色 + 微信云开发 CloudBase + 数据模型可视化后台”的技术路线。

| 层级 | 推荐方案 | 说明 |
| --- | --- | --- |
| 普通用户端 | 继续使用当前原生微信小程序 | 保留现有首页、分类、购物车、订单、拼团等页面，逐步从本地 mock 数据切换到云数据库 |
| 店主端 | 同一个小程序内增加店主入口 | 根据用户 openid 和角色判断是否展示店主端，避免额外开发第二个小程序 |
| 后台管理端 | 使用 CloudBase 数据模型/数据管理能力 | 通过可视化方式管理轮播图、商品、自提点、订单，减少自建 Web 后台成本 |
| 数据层 | 云数据库 | 存储商品、轮播图、自提点、订单、拼团、用户和店主绑定关系 |
| 业务逻辑层 | 云函数 | 处理下单、订单状态变更、店主查询、店主修改自提点、权限校验等敏感逻辑 |
| 文件存储 | 云存储 | 保存商品图片、轮播图图片、分类图标等资源 |

不建议本阶段采用自建服务器、独立后台管理系统或复杂前后端分离架构。原因是项目规模仍以毕业设计和小型生鲜团购演示为主，CloudBase 已能覆盖数据库、云函数、文件存储和基础后台管理需求。

### 22.3 系统角色与权限边界

| 角色 | 主要权限 | 权限边界 |
| --- | --- | --- |
| 普通用户 | 浏览商品、加入购物车、下单、开团、参团、查看自己的订单 | 只能查看和操作自己的订单、地址、收藏和拼团记录 |
| 自提点店主 | 查看本自提点订单、到店商品汇总、收货人信息；修改本自提点基础信息 | 只能访问绑定自提点的数据，不能查看其他自提点订单 |
| 后台管理员 | 管理商品、轮播图、分类、自提点、订单、推荐状态、价格和团购价 | 可通过可视化后台管理平台数据，需要限制管理员账号范围 |

权限控制原则：

- 公开数据如商品、分类、轮播图可允许用户端读取。
- 敏感数据如订单、手机号、收货人地址、店主绑定关系必须通过云函数读取。
- 店主端所有查询均需在云函数中校验当前 openid 是否绑定对应自提点。
- 商品价格、团购价、订单状态等关键字段不允许普通用户前端直接修改。

### 22.4 店主端功能范围

第一版店主端建议控制在 4 个核心页面，优先保证店主能完成自提点履约工作。

#### 22.4.1 店主首页

功能要求：

- 展示当前店主绑定的自提点名称。
- 展示今日待到店订单数、待自提订单数、已提货订单数。
- 展示今日预计到店商品总件数。
- 提供“到店商品”“收货人列表”“自提点设置”入口。

验收标准：

- 店主登录后可看到自己绑定的自提点。
- 店主首页统计数据只包含当前自提点订单。
- 非店主用户不能进入店主首页。

#### 22.4.2 到店商品

功能要求：

- 按商品聚合展示当前自提点的到店商品。
- 展示商品名称、规格、总数量、关联订单数。
- 支持按订单状态筛选，如待发货、待自提、已提货。
- 支持按日期筛选，如今日、明日、全部。

验收标准：

- 同一商品在多个订单中出现时，应合并统计数量。
- 店主可快速判断当天需要准备哪些商品和数量。
- 商品汇总不展示其他自提点的数据。

#### 22.4.3 收货人列表

功能要求：

- 展示当前自提点订单对应的收货人信息。
- 至少展示收货人姓名、手机号、商品明细、订单编号、备注、订单状态。
- 支持按手机号、姓名或订单编号搜索。
- 支持将订单状态从“待自提”更新为“已提货”。

验收标准：

- 店主可查看本自提点订单的收货人信息。
- 店主不能查看其他自提点订单的收货人信息。
- 修改提货状态后，普通用户端订单状态同步变化。

#### 22.4.4 自提点设置

功能要求：

- 店主可查看自提点名称、地址、营业时间、联系电话、公告。
- 店主可修改地址、营业时间、联系电话和公告。
- 自提点名称建议由后台管理员维护，避免店主随意修改造成识别混乱。
- 若后续上线正式运营，可增加管理员审核流程。

验收标准：

- 店主修改自提点信息后，用户端选择自提点和订单确认页能看到更新后的信息。
- 店主只能修改自己绑定的自提点。
- 非店主用户不能访问自提点设置页。

### 22.5 可视化后台管理范围

本项目后台优先使用 CloudBase 可视化数据管理能力，避免从零开发一套复杂 Web 后台。

后台管理员第一版需要管理以下内容：

| 管理模块 | 管理字段 |
| --- | --- |
| 首页轮播图 | 图片、标题、副标题、跳转类型、跳转商品、排序、是否启用 |
| 商品管理 | 商品名称、分类、图片、规格、描述、普通价、原价、团购价、成团人数、库存、限购数量、是否首页推荐、是否支持拼团、上下架状态 |
| 分类管理 | 分类名称、图标、排序、是否启用 |
| 自提点管理 | 自提点名称、地址、营业时间、联系电话、公告、状态、绑定店主 |
| 订单管理 | 订单编号、用户、商品明细、自提点、联系人、手机号、备注、金额、订单状态、下单时间 |
| 拼团管理 | 拼团编号、商品、自提点、团长、成员数、成团人数、状态、创建时间、过期时间、成团时间 |

后台管理原则：

- 首页展示内容由轮播图、商品推荐状态和商品上下架状态共同决定。
- 商品普通价、团购价、库存、是否首页推荐必须从后台配置，不再写死在前端 mock 数据中。
- 订单状态可以由后台管理员兜底修改，但店主端只能处理自己自提点的提货相关状态。
- 后台管理员可新增、下架商品，但普通用户端只展示启用且上架的商品。

### 22.7 云函数规划

建议优先实现以下云函数，统一处理敏感数据和业务规则。

| 云函数 | 用途 |
| --- | --- |
| getOpenId | 获取当前微信用户 openid，当前项目已具备 |
| userApi | 创建/更新用户信息，查询当前用户角色 |
| homeApi | 获取首页轮播图、分类、推荐商品、拼团商品 |
| productApi | 查询商品列表、商品详情、搜索商品 |
| orderApi | 创建订单、查询我的订单、更新用户侧订单状态 |
| groupApi | 开团、参团、查询拼团列表、处理成团逻辑 |
| ownerApi | 店主查询本自提点订单、商品汇总、收货人列表，修改自提点信息 |
| adminApi | 后台补充管理接口，可用于批量初始化数据或特殊管理操作 |

云函数设计原则：

- 下单、开团、参团必须在云函数中重新读取商品价格和库存，不能信任前端传入的价格。
- 参团时必须在云函数中校验拼团状态、剩余名额和库存，避免并发超员。
- 店主查询订单时必须根据当前 openid 反查绑定自提点，只返回允许访问的数据。
- 普通用户查询订单时必须根据当前 openid 过滤，只返回自己的订单。

### 22.8 数据迁移方案

当前项目中的商品、分类、轮播图和自提点主要位于 `utils/mock-data.js`。后续开发时按以下顺序迁移：

1. 将 `categories` 导入 `categories` 集合。
2. 将 `products` 导入 `products` 集合。
3. 将 `banners` 导入 `banners` 集合。
4. 将 `pickupPoints` 导入 `pickup_points` 集合。
5. 保留 `mock-data.js` 作为离线兜底数据，云端读取失败时用于演示。
6. 首页、分类页、详情页优先改为读取云端数据。
7. 订单和拼团从本地缓存迁移到云数据库。

迁移原则：

- 商品编号尽量沿用当前 `grape`、`orange` 等 id，减少页面跳转和收藏数据迁移成本。
- 价格、团购价、库存、是否首页推荐等字段以云数据库为准。
- 订单迁移后，不再依赖本地缓存作为主数据源。

### 22.10 店主端验收清单

- 非店主用户不能看到或进入店主端。
- 店主用户可在“我的”页面进入店主端。
- 店主首页只展示当前店主绑定自提点的数据。
- 到店商品页能按商品汇总展示数量和订单数。
- 收货人列表能展示姓名、手机号、商品明细、备注和订单状态。
- 店主可将本自提点订单从待自提更新为已提货。
- 店主不能查看、修改其他自提点订单。
- 店主可修改本自提点营业时间、联系电话和公告。
- 自提点信息修改后，用户端可看到最新信息。

### 22.11 后台管理验收清单

- 管理员可新增、编辑、启用、停用首页轮播图。
- 管理员可新增、编辑、上下架商品。
- 管理员可设置商品是否首页推荐。
- 管理员可修改商品普通价、原价、团购价、库存和成团人数。
- 管理员可维护自提点名称、地址、营业时间、联系电话和状态。
- 管理员可查看订单列表和订单详情。
- 用户端首页展示内容与后台配置保持一致。
- 商品下架后，用户端不再展示或不能购买该商品。

### 22.12 风险与应对

| 风险 | 影响 | 应对策略 |
| --- | --- | --- |
| 一次性改动过大 | 可能影响现有用户端功能 | 先迁移只读数据，再迁移订单和拼团写入 |
| 数据权限配置错误 | 可能泄露手机号、地址、订单信息 | 敏感数据统一走云函数，前端不直接查询订单集合 |
| 店主误看其他门店订单 | 影响隐私和业务边界 | ownerApi 每次根据 openid 校验自提点绑定 |
| 后台字段过多 | 新手维护困难 | 第一版只保留轮播图、商品、自提点、订单核心字段 |
| 拼团并发超员 | 多人同时参团可能超过 5 人 | 参团逻辑放在云函数中二次校验剩余名额 |
| 云端读取失败 | 演示时页面无数据 | 保留 mock 数据作为演示兜底 |

### 22.13 本阶段结论

本项目下一阶段采用“单小程序多角色 + 微信云开发 CloudBase + 数据模型可视化后台”的方案最合适。该方案能最大程度复用当前原生小程序代码，避免自建服务器和独立后台带来的复杂度，同时满足店主查看到店商品、收货人信息、自提点维护，以及管理员管理轮播图、商品、价格、团购价和订单的核心需求。
