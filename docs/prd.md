## 2. 项目背景

社区生鲜团购以小区、社区、自提点为服务半径，用户可在线浏览生鲜商品、参与团购、加入购物车并提交订单，再到指定自提点取货。相比传统线下买菜，社区团购更适合高频、刚需、价格敏感的生鲜消费场景。

本课题拟建设一个“社区生鲜团购管理小程序”，整体设想包含普通用户端、自提点店主端和后台管理端。当前阶段优先完成普通用户端，使用户能够完成从浏览商品到购物车结算、订单查看与个人中心管理的核心闭环。

## 3. 问题陈述

### 3.1 当前情况

社区用户购买生鲜商品时，经常依赖线下菜场、超市或微信群接龙。线下购买存在时间成本高、商品信息不透明、价格对比不方便等问题；微信群团购又存在商品展示混乱、订单统计困难、用户查询订单不便等问题。

### 3.2 用户痛点

- 商品查找效率低：用户难以快速找到水果、蔬菜、肉蛋、粮油等不同品类。
- 团购信息分散：拼团活动、精选商品、优惠商品缺少统一入口。
- 购物车操作不直观：用户需要方便地选择商品、调整数量、查看总价并结算。
- 订单状态不清晰：用户需要随时查看待发货、待收货、待评价、已完成等订单状态。
- 服务入口不统一：收藏、客服、反馈、退出登录等个人功能需要集中管理。

### 3.3 业务影响

- 提升社区用户购买生鲜商品的效率和体验。
- 提高团购商品曝光率，促进拼团抢购和今日精选商品转化。
- 规范购物车与订单流程，降低订单遗漏和沟通成本。
- 为后续自提点店主端、后台管理端提供清晰的业务边界。

## 4. 产品目标

### 4.1 核心目标

普通用户端需要支持用户完成以下核心流程：

1. 进入首页浏览推荐商品、商品分类、拼团抢购与今日精选。
2. 通过搜索或分类快速找到目标商品。
3. 查看商品信息并加入购物车。
4. 在购物车中选择商品、调整数量、查看总价并发起结算。
5. 在“我的”页面查看订单状态、收藏、客服、反馈和账号相关功能。

### 4.2 成功指标

- 用户可在 3 次点击以内进入任意一级商品分类。
- 用户可在购物车页面直接完成勾选、数量调整、总价确认和结算操作。
- 首页能清晰呈现搜索、轮播图、分类入口、拼团抢购和今日精选模块。
- “我的”页面能清晰呈现订单状态入口和常用个人服务入口。
- 主要页面视觉风格统一，符合生鲜团购绿色、清爽、便捷的产品感受。

## 5. 用户角色

### 5.1 普通用户

普通用户是本阶段的主要使用者，通常为社区居民、家庭采购者或日常生鲜消费者。

核心诉求：

- 快速发现新鲜、优惠、适合家庭采购的商品。
- 清楚了解商品价格、图片、名称、规格等信息。
- 能方便地加入购物车并合并结算。
- 能查看订单进度和个人服务入口。

### 5.2 自提点店主

自提点店主是后续阶段角色，本阶段仅作为业务边界存在。普通用户端需要让用户理解订单最终将由自提点或平台完成履约，但不定义店主端具体功能。

### 5.3 后台管理员

后台管理员是后续阶段角色，本阶段仅作为商品、订单、轮播图、分类等内容的管理来源，不定义后台管理端具体功能。

## 8. 核心业务流程

### 8.1 浏览购买流程

1. 用户进入小程序首页。
2. 用户通过搜索框、分类入口、拼团抢购或今日精选进入商品列表或商品详情。
3. 用户查看商品信息。
4. 用户将商品加入购物车。
5. 用户进入购物车，选择需要购买的商品。
6. 用户调整商品数量并确认合计金额。
7. 用户点击结算，进入订单确认流程。
8. 用户提交订单后，可在“我的”页面查看订单状态。

### 8.2 购物车管理流程

1. 用户进入购物车。
2. 页面展示已加入购物车的商品。
3. 用户可单选、多选或全选商品。
4. 用户可增加或减少商品数量。
5. 页面根据已选商品实时展示合计金额。
6. 用户可删除不需要的商品。
7. 用户点击结算进入下一步。

### 8.3 个人中心流程

1. 用户进入“我的”页面。
2. 页面展示用户头像、昵称和订单状态入口。
3. 用户点击订单状态查看对应订单。
4. 用户可进入我的收藏、在线客服、意见反馈。
5. 用户可执行退出登录。

## 9. 功能需求

### 9.1 底部 TabBar

底部导航包含“首页”“分类”“购物车”“我的”四个入口。

功能要求：

- 当前选中的 Tab 需要有明显高亮状态。
- 未选中的 Tab 以弱化样式展示。
- 用户点击任一 Tab 后，应切换到对应页面。
- Tab 顺序固定为：首页、分类、购物车、我的。
- 视觉风格参考截图，主色调以绿色为核心，强调生鲜、健康、便捷。

验收标准：

- 用户能从任意一级页面切换到其他 Tab 页面。
- 当前页面对应 Tab 的图标和文字均有选中状态。
- 四个 Tab 名称与顺序保持一致。

### 9.2 首页

首页是普通用户端的主要流量入口，布局参考图 1，自上而下包含商品搜索框、轮播图、商品简单分类、拼团抢购、今日精选。

#### 9.2.1 顶部区域

功能要求：

- 页面顶部展示小程序标题，建议名称为“农产品拼团商城”或与课题一致的产品名称。
- 顶部主色为绿色，整体保持清爽、醒目的视觉风格。
- 顶部区域下方展示商品搜索框。

验收标准：

- 用户进入首页后可立即看到标题和搜索框。
- 搜索框占据页面上方主要视觉位置。

#### 9.2.2 商品搜索框

功能要求：

- 搜索框提示文案建议为“请输入商品关键词”。
- 用户点击搜索框后，可输入商品名称、分类关键词或商品相关词。
- 提交搜索后展示与关键词相关的商品结果。
- 当搜索结果为空时，需要给出空状态提示。

验收标准：

- 输入有效关键词后能进入搜索结果页或展示搜索结果。
- 输入无匹配关键词时能看到明确提示。
- 搜索框在首页顶部固定位置清晰可见。

#### 9.2.3 轮播图

功能要求：

- 轮播图用于展示生鲜团购活动、主推商品、节日促销或平台公告。
- 轮播图应位于搜索框下方。
- 轮播图支持多张图片轮换展示。
- 用户点击轮播图，可进入对应活动或商品详情。
- 轮播图下方展示当前轮播位置指示。

验收标准：

- 首页轮播图能正常展示至少一张图片。
- 多张轮播内容之间可切换。
- 点击轮播图能进入对应内容页面。

#### 9.2.4 商品简单分类

首页分类区参考图 1，以图标加文字形式展示常用分类。

建议分类：

- 新鲜水果
- 时令蔬菜
- 禽畜肉类
- 蛋品采购
- 粮油调味
- 新鲜乳品
- 五谷杂粮
- 干货菌菇

功能要求：

- 分类以宫格方式展示，建议每行四个。
- 每个分类包含分类图标和分类名称。
- 用户点击分类后进入对应分类商品列表。
- 分类名称需要简短、易懂，避免换行过多。

验收标准：

- 首页能展示不少于 8 个常用分类入口。
- 点击任一分类后能看到对应分类商品。
- 分类样式在页面中整齐排列。

#### 9.2.5 拼团抢购

拼团抢购用于展示限时或重点团购商品。

功能要求：

- 拼团抢购模块位于首页分类区下方。
- 模块标题为“拼团抢购”。
- 商品卡片展示商品图片、名称、价格、团购标识或活动信息。
- 用户点击商品卡片进入商品详情。
- 商品售罄或活动结束时，需要展示对应状态。

验收标准：

- 首页能看到“拼团抢购”模块标题。
- 模块内至少可展示一个商品卡片。
- 商品卡片信息能帮助用户判断是否购买。

#### 9.2.6 今日精选

今日精选用于展示平台推荐商品、应季商品或高性价比商品。

功能要求：

- 模块标题为“今日精选”。
- 商品卡片展示图片、名称、价格、销量或推荐理由。
- 用户点击商品卡片进入商品详情。
- 今日精选可与拼团抢购形成区分，避免内容重复感过强。

验收标准：

- 首页能看到“今日精选”模块。
- 商品信息展示完整清晰。
- 点击商品后能进入商品详情或商品购买路径。

### 9.3 分类页

分类页用于系统化展示所有商品品类，帮助用户按需求浏览商品。

功能要求：

- 分类页展示一级商品分类。
- 用户选择某一分类后，展示该分类下的商品列表。
- 商品列表应展示商品图片、名称、价格、简要描述和购买入口。
- 分类页应支持从分类进入商品详情。
- 当某分类暂无商品时，应展示空状态提示。

验收标准：

- 用户能通过分类页浏览不同商品品类。
- 切换分类后商品列表内容随之变化。
- 商品卡片信息清晰，不影响用户判断。

### 9.4 商品详情页

商品详情页用于承载用户购买决策。

功能要求：

- 展示商品图片、商品名称、价格、规格、库存状态、商品描述。
- 展示商品是否参与拼团、是否为精选商品等活动信息。
- 用户可选择购买数量。
- 用户可将商品加入购物车。
- 用户可直接进入结算流程。
- 用户可收藏商品。
- 当商品售罄或不可购买时，购买入口需要置灰或提示不可购买原因。

验收标准：

- 用户能从首页、分类页、搜索结果或购物车进入商品详情。
- 商品详情信息完整，能支撑用户购买决策。
- 加入购物车后购物车数量和内容应更新。

### 9.5 购物车页

购物车页布局与功能参考图 2，主要包含商品列表、选择状态、数量调整、合计金额和结算入口。

#### 9.5.1 商品列表

功能要求：

- 展示用户已加入购物车的商品。
- 每个商品行展示选择按钮、商品图片、商品名称、商品价格、购买数量、增加按钮、减少按钮。
- 商品名称过长时需要合理展示，不应影响价格和数量操作。
- 商品失效、售罄或库存不足时，需要给出提示并限制结算。

验收标准：

- 购物车有商品时能按列表展示商品。
- 商品图片、名称、价格、数量、选择状态清晰可见。
- 商品不可购买时有明确状态提示。

#### 9.5.2 商品选择

功能要求：

- 用户可勾选或取消勾选单个商品。
- 页面底部提供“全选”入口。
- 全选后所有可购买商品均被选中。
- 取消全选后所有商品取消选中。

验收标准：

- 单个商品选择状态可正常切换。
- 全选状态与商品选择状态保持一致。
- 不可购买商品不应参与结算合计。

#### 9.5.3 数量调整

功能要求：

- 用户可点击加号增加商品数量。
- 用户可点击减号减少商品数量。
- 数量不能小于 1。
- 数量达到库存上限或购买限制时，应提示用户。
- 数量变化后，合计金额应同步更新。

验收标准：

- 点击加号后商品数量增加。
- 点击减号后商品数量减少。
- 数量变化后合计金额正确变化。
- 数量达到限制时有清晰提示。

#### 9.5.4 合计与结算

功能要求：

- 底部固定展示全选、合计金额和结算按钮。
- 合计金额只计算已选中的可购买商品。
- 合计金额格式清晰，突出价格信息。
- 未选择商品时，结算按钮应不可用或点击后提示选择商品。
- 点击结算后进入订单确认流程。

验收标准：

- 勾选商品后合计金额正确展示。
- 未勾选商品时不能直接结算。
- 点击结算能进入订单确认流程。

#### 9.5.5 空购物车

功能要求：

- 当购物车为空时，展示空状态。
- 空状态应提供返回首页或去逛逛入口。

验收标准：

- 无商品时不展示空白页面。
- 用户能从空购物车快速回到商品浏览路径。

### 9.6 订单确认页

订单确认页用于用户提交订单前的最终确认。

功能要求：

- 展示待购买商品列表、数量、单价和小计。
- 展示订单总金额。
- 展示自提点或取货信息入口。
- 展示用户联系方式入口。
- 支持用户填写订单备注。
- 用户确认无误后可提交订单。

验收标准：

- 用户能确认商品、数量、金额和取货信息。
- 必填信息缺失时不能提交订单，并给出提示。
- 订单提交成功后给出成功反馈，并引导用户查看订单。

### 9.7 订单状态

订单状态入口主要在“我的”页面展示，参考图 3。

功能要求：

- 展示四个订单状态入口：待发货、待收货、待评价、已完成。
- 用户点击某个状态后，进入对应订单列表。
- 订单列表展示订单商品、订单金额、订单状态、下单时间、操作入口。
- 不同状态下展示不同可用操作，例如确认收货、去评价、查看详情。

验收标准：

- “我的”页面能清晰看到四个订单状态。
- 点击状态后能进入对应订单列表。
- 空订单状态需要展示友好的空状态提示。

### 9.8 我的页面

我的页面布局与功能参考图 3，主要承载个人信息、订单入口和常用服务。

#### 9.8.1 用户信息

功能要求：

- 页面顶部展示用户头像和昵称。
- 未登录状态下应引导用户登录或授权。
- 登录后展示用户昵称。

验收标准：

- 用户进入“我的”页面后能看到个人信息区域。
- 未登录用户能看到登录引导。

#### 9.8.2 订单快捷入口

功能要求：

- 展示待发货、待收货、待评价、已完成四个入口。
- 每个入口使用图标加文字形式展示。
- 如存在对应状态订单，可展示数量提醒。

验收标准：

- 四个订单入口名称准确。
- 点击入口后进入对应订单列表。
- 订单状态图标视觉统一。

#### 9.8.3 我的收藏

功能要求：

- 用户可进入我的收藏列表。
- 收藏列表展示用户收藏过的商品。
- 用户可从收藏列表进入商品详情。
- 用户可取消收藏。

验收标准：

- 用户能查看已收藏商品。
- 取消收藏后列表状态更新。

#### 9.8.4 在线客服

功能要求：

- 用户可通过在线客服入口联系平台或商家。
- 客服入口需要放在“我的”页面常用服务区域。

验收标准：

- 点击在线客服后能进入客服沟通或客服引导页面。
- 若客服不可用，需要展示替代联系方式或提示。

#### 9.8.5 意见反馈

功能要求：

- 用户可提交使用问题、商品建议、服务建议或其他反馈。
- 反馈内容应支持文字描述。
- 提交成功后需要给出反馈成功提示。

验收标准：

- 用户能进入反馈页面并填写内容。
- 空内容不能提交。
- 提交后用户能看到明确结果反馈。

#### 9.8.6 退出登录

功能要求：

- 用户可点击退出登录。
- 退出前需要二次确认。
- 退出后回到未登录状态。

验收标准：

- 点击退出登录后出现确认提示。
- 确认退出后个人信息变为未登录状态。

## 10. 页面视觉与交互要求

### 10.1 视觉风格

- 主色调以绿色为主，传达生鲜、健康、自然、便捷的品牌感。
- 背景以白色或浅色为主，突出商品图片与价格。
- 价格信息使用醒目颜色，便于用户快速识别。
- 页面整体应保持清爽，不堆叠过多装饰元素。
- 图标风格统一，分类图标应与商品品类强相关。

### 10.2 布局要求

- 首页布局顺序为：顶部标题、搜索框、轮播图、商品分类、拼团抢购、今日精选。
- 购物车底部结算栏固定展示，便于用户随时结算。
- “我的”页面订单状态入口应位于用户信息下方，服务入口按列表排列。
- 页面内容需要适配常见手机屏幕，不应出现文字遮挡或按钮难以点击。

### 10.3 交互要求

- 所有可点击元素需要有清晰的点击反馈。
- 商品加入购物车、提交订单、收藏、取消收藏等关键操作需要结果提示。
- 删除、退出登录等可能造成状态变化的操作需要确认。
- 空状态、加载状态、错误状态需要有明确文案。

## 11. 内容与数据展示要求

### 11.1 商品基础信息

商品展示至少应包含：

- 商品图片
- 商品名称
- 商品价格
- 商品分类
- 商品规格或简要描述
- 库存或可购买状态
- 是否参与拼团或精选推荐

### 11.2 订单基础信息

订单展示至少应包含：

- 订单编号或订单标识
- 商品信息
- 购买数量
- 订单金额
- 订单状态
- 下单时间
- 自提或取货相关信息

### 11.3 用户基础信息

用户展示至少应包含：

- 头像
- 昵称
- 登录状态
- 收藏记录入口
- 订单状态入口

## 12. 用户故事与验收标准

### 12.1 首页浏览

用户故事：

作为普通用户，我希望进入小程序后可以快速看到搜索、活动、分类和推荐商品，以便快速找到想买的生鲜商品。

验收标准：

- 首页首屏可看到搜索框、轮播图和分类入口。
- 用户可通过分类、拼团抢购、今日精选进入商品购买路径。

### 12.2 商品搜索

用户故事：

作为普通用户，我希望通过关键词搜索商品，以便快速找到目标商品。

验收标准：

- 搜索框支持输入关键词。
- 搜索后展示相关商品。
- 无结果时展示空状态。

### 12.3 分类浏览

用户故事：

作为普通用户，我希望按照水果、蔬菜、肉类、蛋品等分类浏览商品，以便提高挑选效率。

验收标准：

- 分类入口清晰可见。
- 点击分类后展示对应商品列表。
- 商品列表可进入商品详情。

### 12.4 加入购物车

用户故事：

作为普通用户，我希望在浏览商品时能把想买的商品加入购物车，以便统一结算。

验收标准：

- 商品详情页提供加入购物车入口。
- 加入成功后给出提示。
- 购物车页面能看到新增商品。

### 12.5 购物车结算

用户故事：

作为普通用户，我希望在购物车中勾选商品、调整数量并查看合计金额，以便确认购买内容后结算。

验收标准：

- 支持单选、多选、全选。
- 支持数量增加和减少。
- 合计金额随选择和数量变化更新。
- 点击结算进入订单确认流程。

### 12.6 查看订单

用户故事：

作为普通用户，我希望在个人中心按状态查看订单，以便了解购买进度。

验收标准：

- “我的”页面展示待发货、待收货、待评价、已完成。
- 点击任一状态后展示对应订单列表。
- 无订单时展示空状态。

### 12.7 使用个人服务

用户故事：

作为普通用户，我希望在“我的”页面找到收藏、客服、反馈和退出登录入口，以便管理个人信息和服务需求。

验收标准：

- “我的”页面展示我的收藏、在线客服、意见反馈、退出登录。
- 点击每个入口能进入对应功能。
- 退出登录前有确认提示。

## 13. 业务规则

### 13.1 商品规则

- 商品应归属到明确分类。
- 商品不可购买时，应明确展示状态。
- 商品价格需要在列表、详情、购物车和订单中保持一致。
- 拼团商品和普通商品需要在展示上有所区分。

### 13.2 购物车规则

- 购物车只展示用户已加入的商品。
- 购物车合计金额只计算已选商品。
- 不可购买商品不参与结算。
- 商品数量不能小于 1。
- 当商品数量达到限制时，继续增加应提示用户。

### 13.3 订单规则

- 提交订单前必须确认商品、数量、金额和取货信息。
- 订单提交后进入待发货状态。
- 用户确认收货后，订单可进入待评价状态。
- 用户评价完成或流程结束后，订单进入已完成状态。

### 13.4 账号规则

- 未登录用户可浏览商品。
- 涉及加入购物车、收藏、提交订单、查看订单等个人行为时，应引导登录。
- 退出登录后，不再展示当前用户的个人信息。

## 14. 异常与空状态

| 场景 | 展示要求 |
| --- | --- |
| 首页无轮播内容 | 展示默认图片或隐藏轮播区域，页面布局保持自然 |
| 分类无商品 | 展示“暂无相关商品”类提示 |
| 搜索无结果 | 展示“未找到相关商品”类提示，并提供返回首页或换词搜索 |
| 购物车为空 | 展示空购物车提示，并提供去首页逛逛入口 |
| 商品售罄 | 商品展示售罄状态，限制购买 |
| 网络或加载异常 | 展示加载失败提示，并提供重试入口 |
| 未登录访问个人功能 | 引导用户登录 |

## 15. 质量要求

- 页面结构清晰，用户能理解每个区域的用途。
- 关键购买流程完整，不出现流程中断。
- 页面跳转路径明确，用户能从任意核心页面回到首页或底部 Tab。
- 价格、数量、订单状态等关键信息展示准确。
- 表单提交、删除、退出登录等操作需要有明确反馈。
- 常见空状态和异常状态不应出现无提示的空白页面。

## 16. 毕设验收建议

### 16.1 基础验收

- 四个底部 Tab 页面均可进入。
- 首页包含搜索框、轮播图、分类、拼团抢购、今日精选。
- 分类页可展示分类和商品列表。
- 购物车可完成商品选择、数量调整、合计与结算入口展示。
- “我的”页面包含头像昵称、订单状态、收藏、客服、反馈、退出登录。

### 16.2 核心流程验收

- 从首页进入商品详情，再加入购物车。
- 从购物车选择商品并进入订单确认。
- 提交订单后可在“我的”页面查看对应订单状态。
- 收藏商品后可在“我的收藏”中查看。
- 未登录状态访问个人功能时有登录引导。

### 16.3 展示验收

- 页面风格与参考图保持一致的绿色生鲜商城风格。
- 商品图片、名称、价格、数量、按钮展示清晰。
- 底部导航、购物车结算栏、个人中心入口布局清楚。

## 18. 风险与应对

| 风险 | 影响 | 应对策略 |
| --- | --- | --- |
| 需求范围过大 | 毕设周期内难以完成 | 优先完成普通用户端核心闭环，店主端和后台端作为后续扩展 |
| 生鲜商品状态变化频繁 | 价格、库存、售罄状态容易影响购买体验 | 在商品列表、详情、购物车中明确展示商品状态 |
| 购物车规则不清晰 | 用户可能误解合计金额或结算范围 | 明确区分选中商品和未选中商品，突出合计金额 |
| 拼团概念理解成本高 | 用户不清楚拼团商品与普通商品差异 | 在拼团抢购模块和商品详情中展示团购标识与说明 |
| 页面入口过多 | 用户可能找不到关键功能 | 保持四个 Tab 清晰，常用功能优先展示 |
| 订单状态不完整 | 演示时难以体现购买闭环 | 至少保留待发货、待收货、待评价、已完成四个状态 |

## 19. 后续扩展方向

普通用户端完成后，可继续规划：

- 自提点店主端：订单接收、到货提醒、用户提货核销、订单统计。
- 后台管理端：商品管理、分类管理、轮播图管理、订单管理、用户管理、数据统计。
- 营销能力：优惠券、限时抢购、满减活动、会员积分。
- 服务能力：售后申请、退款处理、消息通知、评价管理。

## 20. 总结

普通用户端是“社区生鲜团购管理小程序”的核心入口，本期应优先保证用户能够完成浏览商品、查找商品、加入购物车、结算下单、查看订单和管理个人服务的完整路径。页面设计需贴合参考图中的绿色生鲜商城风格，重点突出商品、价格、分类、购物车和订单状态，形成清晰、轻量、适合毕业设计演示的产品闭环。
